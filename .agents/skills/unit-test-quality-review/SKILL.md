# Проверка качества тестов

## Что делают хорошие тесты

Хорошие тесты:
- Защищают от багов (ловят регрессии)
- Устойчивы к рефакторингу (не ломаются при изменении реализации)
- Дают быструю обратную связь (не делают лишней работы)
- Просты в поддержке (читаются за секунды)

## Что делает тесты плохими

### 1. Тестирование приватных методов напрямую

Избегай тестирования private/protected методов как таковых.
Тестируй публичный API — косвенно он проверит и приватную логику.

**Исключение:** приватный метод содержит сложную алгоритмическую логику
(например, парсинг, расчёт, валидацию) и вызывается только через публичный
метод, которому для проверки всех кейсов нужно слишком много окружения.
В этом случае можно протестировать его как утилиту, но через рефлексию
или сделав package-private/internal.

### 2. Раскрытие приватного состояния ради теста

Никогда не делай поля/public/protected только для того, чтобы их проверить
в тесте. Если хочется проверить внутреннее состояние — значит:
- ты тестируешь реализацию, а не поведение
- нужен либо публичный метод, который показывает это состояние
- либо тест нужно переписать на проверку результата

### 3. Дублирование доменной логики в тесте

Если тест содержит ту же логику фильтрации/расчёта/трансформации, что и
продакшен-код, то при ошибке в продакшене тест тоже "пройдёт", потому что
сделает ту же ошибку.

Вместо этого: захардкодь ожидаемый результат.
- ❌ `assert result.filter(x => x.active).map(x => x.name) == items.filter(x => x.active).map(x => x.name)`
- ✅ `assert result == [item1, item3]`

### 4. Моки конкретных классов вместо интерфейсов

Мокай интерфейсы/абстракции/границы системы, а не конкретные реализации.

- ❌ мок `new UserService()` с конкретной логикой
- ✅ мок `interface IEmailSender` или `EmailSenderPort`

Это позволяет менять реализацию без правки тестов.

### 5. Управляющая логика в теле теста

В теле теста не должно быть `if`, `for`, `while`, `try/catch`.

Если они есть — значит тест делает слишком много или проверяет несколько
вещей. Разбей на отдельные тесты.

`try/catch` в тесте допустим только если сам тестируемый метод может
бросить исключение, и ты проверяешь именно это (и то — лучше использовать
встроенный assertion для эксепшенов).

### 6. Несколько проверок в одном тесте

Один тест — одна проверка. Если тест валится, ты должен сразу понять,
какое именно ожидание нарушено, по названию теста и первой же упавшей
строчке.

## Типы проверок (в порядке предпочтения)

1. **Проверка выходных данных (output)** — самый ценный тип:
   вызвал метод, проверил что вернулось.

2. **Проверка состояния (state)** — проверил что изменилось в
   БД/хранилище/объекте после вызова.

3. **Проверка взаимодействия (interaction)** — проверил что мок
   был вызван с ожидаемыми аргументами. Наименее ценный тип,
   т.к. привязывает тест к реализации.

Старайся использовать проверки как можно более высокого уровня.
Начинай с output, если не получается — state, и только в последнюю
очередь interaction.

## Формат ответа

При проверке тестов указывай конкретные проблемы с ссылками на код
(файл:строка) и с пояснением, почему это проблема и как исправить.
